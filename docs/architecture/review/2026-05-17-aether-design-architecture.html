Aether Architecture Review

Phase 1 design review — multi-agent orchestration framework over the Envelope IPC protocol

Generated 2026-05-17 · Design Review

Executive Summary

Aether is an independent Rust orchestration framework that composes AgentVerse single-agent processes into multi-agent DAG workflows via a neutral Envelope IPC protocol (newline-delimited JSON over stdio or Unix sockets). The architecture makes strong use of the Ports and Adapters pattern — the Transport trait cleanly abstracts IPC from orchestration logic, enabling a Phase 2 swap to TCP without touching the DAG engine. The primary areas to address before implementation are: bounding the Singleton request queue, adding a timeout mechanism to Transport::send(), splitting the Supervisor's responsibilities, and adding auth hooks to the dashboard ahead of Phase 2 management endpoints.

Architecture

Strength
Ports and Adapters — Transport trait

The Transport trait is a textbook port. StdioTransport and UnixSocketTransport are adapters, and Phase 2's TcpTransport is a third. The DAG engine never knows how agents are contacted — it only calls send(Envelope). Swapping transports requires zero changes to orchestration logic.

Strength
Strong anti-corruption layer — zero shared code

Aether and AgentVerse share no code. The Envelope protocol is the only contract, implemented independently on each side. This is a clean anti-corruption layer that lets both projects evolve without coupling.

Strength
Separation of concerns — aether-core vs aether-dashboard

Orchestration logic and presentation are entirely separate crates. The dashboard subscribes to the Supervisor's event stream and has no direct access to workflow or registry internals. Changes to the dashboard cannot affect orchestration correctness.

Concern
Supervisor is approaching a god service

The Supervisor owns instance lifecycle, FailurePolicy enforcement, health check Ping/Pong, and the watch() event stream broadcast. That is four distinct responsibilities in one component. Consider extracting an InstanceManager for process lifecycle so the Supervisor focuses on policy and observability.

Concern
Registry mutation during live workflows

AgentRegistry supports hot-swap via Arc<RwLock>. The spec doesn't define what happens to in-flight workflow instances when their node definition is replaced mid-execution. An in-flight dispatch could race with a registry update, leading to inconsistent behavior within a single workflow run.

Concern
Fan-in payload contract is implicit

Fan-in collects branch results as a JSON array and passes it as the downstream node's payload. The downstream agent must know to expect an array rather than a single value. This is an implicit contract between the workflow author and agent implementation with no enforcement mechanism.

Security

Strength
No network surface in Phase 1

All agent communication in Phase 1 is over stdio pipes or Unix domain sockets — both local OS IPC mechanisms with no network exposure. The attack surface is limited to local process communication, which is a significantly smaller risk than an HTTP API.

Risk
No authentication or authorization defined

The dashboard and workflow.run() entry point have no authentication mechanism. Phase 2 management endpoints (restart, pause, hot-swap agents) over a potentially network-accessible dashboard without auth would be a critical vulnerability. Auth hooks should be designed into the axum server now, even if disabled by default in Phase 1.

Concern
Envelope metadata is agent-controlled and unvalidated

Envelope metadata is a HashMap<String, String> written by the agent process. A buggy or malicious agent could set workflow_id, trace_id, or other fields to arbitrary values that confuse the Supervisor or poison observability data. Aether should treat agent-supplied metadata as untrusted and validate or namespace it on receipt.

Concern
Unix socket file permissions unspecified

The spec doesn't address what permissions are set on Unix socket files created by UnixSocketTransport. A world-readable socket allows any local process to inject Envelopes. Socket files should be created with 0o600 (owner-only) by default.

Concern
Untyped payload — no validation boundary

The Envelope payload is serde_json::Value with no schema validation. A malformed or malicious payload from one agent is forwarded verbatim to the next. Consider validating payload structure at node boundaries, at minimum checking that the value is an object before forwarding.

Scalability

Strength
Fully async execution — Tokio throughout

Fan-out branches spawn concurrent Tokio tasks, and fan-in uses JoinSet. The architecture handles many parallel workflows without blocking OS threads. This is a strong scalability foundation for a single-process deployment.

Strength
Phase 2 horizontal scaling seams are explicit

The Transport trait, AgentFactory, and AgentRegistry discovery layer are all called out as Phase 2 extension points. Scaling out to multiple machines is an implementation swap, not an architectural change.

Concern
Singleton queue is unbounded

SpawnPolicy::Singleton queues requests with "never rejected" semantics. Under sustained load this queue can grow without bound, exhausting memory. A max_queue_depth: Option<usize> field on Singleton (or a backpressure signal to the caller) should be added.

Concern
SupervisorEvent broadcast — slow consumer behavior undefined

The watch() stream is broadcast to all subscribers. If the dashboard SSE consumer falls behind (e.g., a slow browser connection), the behavior is undefined — events may be dropped or the broadcast channel may apply backpressure to the Supervisor itself. The buffering strategy and slow-consumer policy should be specified.

Concern
Health check interval has no bounds

The Ping/Pong health check interval is described as "configurable" with no defaults, minimums, or maximums specified. A misconfigured short interval floods agents with health checks; a long interval means slow failure detection. Sensible defaults (e.g., 30s interval, 3 missed Pongs = unhealthy) should be defined.

Reliability

Strength
Declarative per-node FailurePolicy

Retry count, restart-on-failure, and fallback routing are all first-class, per-node configuration. Workflow authors have explicit control over failure behavior at each node without writing recovery code in the orchestration layer.

Strength
Typed Outcome — no stranded errors

Outcome::Success | Failed | Timeout gives callers structured, matchable failure information. Callers know which node failed and why, enabling intelligent application-level recovery beyond what the FailurePolicy handles.

Concern
No timeout on Transport::send()

Transport::send() has no timeout parameter or deadline. A hanging agent process (e.g., model server stalled mid-generation) blocks a Tokio task indefinitely. AetherError::AgentTimeout exists but the mechanism that triggers it is unspecified. A per-node timeout: Duration in AgentNode or FailurePolicy is needed.

Concern
PerRequest orphan process risk

For PerRequest agents, the process is dropped after receiving Result. If the agent does not exit cleanly (e.g., it is waiting on I/O), the child process becomes an orphan. Over many requests this accumulates zombie processes. The spec should define how Aether ensures process cleanup — SIGTERM with timeout, then SIGKILL.

Concern
No circuit breaker pattern

If a Pool of agents is consistently failing (e.g., the underlying model server is down), Aether will retry N times per request × M concurrent requests in parallel. This amplifies load on an already-failing system. A simple in-process circuit breaker per AgentNode — open after K consecutive failures, half-open after a cool-down — would prevent this cascade.

Risk
Single Supervisor is a single point of failure

The Supervisor is the sole runtime owner of all live agent instances and active workflow state. If the Aether process crashes, all in-flight workflows and agent connections are lost with no recovery path. This is acceptable for Phase 1 (no clustering), but the spec should explicitly note that workflow state is ephemeral and that callers must handle Outcome::Failed for crash recovery.

Recommendations

  1. Add timeout to AgentNode or FailurePolicy — Introduce timeout: Duration to AgentNode (or FailurePolicy). The DAG engine wraps each Transport::send() in tokio::time::timeout(), returning AetherError::AgentTimeout on expiry. This activates the existing timeout variant and prevents hung tasks.
  2. Split Supervisor into InstanceManager + Supervisor — Extract process spawn, health Ping/Pong, and lifecycle teardown into an InstanceManager. Keep FailurePolicy enforcement and the watch() event stream in Supervisor. Each component then has one clear responsibility.
  3. Add auth hooks to the dashboard now — Even if disabled by default in Phase 1, add an optional token middleware to the axum server (e.g., --auth-token <secret> flag that enables Bearer token checking on all routes). Phase 2 management endpoints need this and retrofitting auth is significantly harder than leaving the hook.
  4. Bound the Singleton request queue — Add max_queue_depth: Option<usize> to SpawnPolicy::Singleton. When the queue is full, return Err(AetherError::WorkflowError) immediately rather than queuing indefinitely. Default to None (unbounded) but document the risk.
  5. Define PerRequest process cleanup — Specify that on Result receipt, Aether sends SIGTERM to the child process, waits up to a configurable grace period (e.g., 5s), then sends SIGKILL. Add a shutdown_grace_ms: u64 field to FailurePolicy or AgentNode.
  6. Namespace agent-supplied Envelope metadata — On receipt of a Result or Error Envelope, Aether should copy only known metadata keys (model, provider, tokens_input, tokens_output) from the agent-supplied metadata into the internal Envelope, discarding everything else. This prevents agents from spoofing trace_id or workflow_id.
  7. Specify Unix socket permissions — UnixSocketTransport should create socket files with 0o600 permissions by default (owner read/write only). Document this in the spec and make it configurable for scenarios where group access is intentional.
  8. Define fan-in payload contract explicitly — Either standardize the fan-in merge as a JSON array (document in spec), or introduce a FanInReducer trait that defaults to array-merge but allows custom reduction. This removes the implicit contract between workflow author and downstream agent.
